# /write-restaurant — 맛집 블로그 글 작성

`rules/structure.md`, `rules/tone.md`, `rules/html_template.md`, `rules/examples.md` 4개 파일을 **반드시 먼저 모두 읽고** 그 톤·구조·템플릿을 정확히 따른다.

## 진행 순서

### Step 1 — 협찬 여부 확인 (가장 먼저, 사진 받기 전)

`AskUserQuestion` 으로 다음을 묻는다:

- 질문: "이 글이 협찬·제공·체험단인가요?"
- header: "협찬 여부"
- options:
  1. label: "내돈내산 (직접 결제)" — description: "본문에 #내돈내산 마커가 들어갑니다"
  2. label: "협찬·체험단·제공" — description: "#내돈내산 절대 안 넣음. 협찬 명시 문구를 본문에 넣어야 합니다"

사용자가 **협찬·체험단·제공** 을 선택하면, 이어서 자유 답으로 묻는다:
- "본문에 넣을 협찬 명시 문구를 알려주세요. (예: '소정의 원고료를 받고 작성된 글입니다' 또는 '○○ 브랜드로부터 제품을 제공받아 작성한 후기입니다'). 그냥 '기본 문구로'라고 하시면 표준 문구를 넣을게요."

이 정보(`is_sponsored`, `sponsor_disclosure`)를 메모리에 보관하고 Step 5/6에서 사용한다.

### Step 1.5 — 최소 글자수 확인

협찬에는 흔히 "1000자 이상", "1500자 이상" 같은 분량 조건이 붙는다. 또한 사용자가 그냥 자세히 쓰고 싶을 때도 있다. `AskUserQuestion` 으로 묻는다:

- 질문: "이 글의 최소 글자수 조건이 있나요? (공백 포함, HTML 태그·주석·해시태그 제외)"
- header: "최소 글자수"
- options:
  1. label: "1000자 이상" — description: "표준적인 협찬 분량"
  2. label: "1500자 이상" — description: "조금 더 자세한 협찬"
  3. label: "2000자 이상" — description: "장문 후기 / 까다로운 협찬"
  4. label: "분량 제한 없음" — description: "정보가 있는 만큼 자연스럽게"

사용자가 **Other** 로 직접 숫자를 적으면 그 값을 사용한다 (예: `1200`, `2500`).

이 정보(`min_chars`)를 메모리에 보관하고 Step 5에서 글이 그 분량을 넘도록 작성한다. Step 6에서 실제 글자수를 측정해 사용자에게 보고한다.

### Step 2 — 사진 요청

다음 메시지를 출력:

```
이제 가게 사진들을 채팅에 끌어다 놓아주세요. 🍽
권장 5~15장 (외관 / 매장 / 메뉴판 / 음식 클로즈업 / 단면컷 등)
```

그리고 사용자의 다음 메시지를 기다린다.

### Step 3 — 사진 받으면 자동 인식

사용자가 사진을 첨부하면 각 사진을 보고 다음을 식별:
- 음식 종류 (햄버거, 파스타, 한식 등)
- 외관 / 메뉴판 / 매장 / 음식 클로즈업 등 분류
- 메뉴판이 보이면 메뉴명·가격을 텍스트로 추출
- 매장 분위기 형용사 (밝음, 클래식, 좁음/넓음 등)

식별 결과를 사용자에게 1~2문장으로 요약 (예: "햄버거집이고 메뉴판 사진도 있네요. 모아나버거 / 하와이안버거 / 감자튀김 세트로 보여요").

### Step 4 — 추가 정보 질문

`AskUserQuestion` 으로 한꺼번에 묻는다 (필요한 항목만, 사진에서 이미 명확하면 생략):

1. **가게 이름** (텍스트 입력 필요 시 자유 답)
2. **위치** — 가까운 지하철역 + 출구 + 도보 분 (예: "합정역 7번 출구 도보 5분")
3. **방문 일자** — 평일/주말, 점심/저녁
4. **추천 대상** — 데이트 / 회식 / 가족모임 / 혼밥 / 가성비 등 1~2개
5. **한 줄 총평** — 맛/분위기 핵심 인상 (자유 답)

`AskUserQuestion` 에 들어가지 않는 자유 텍스트는 일반 질문으로 (가게명, 정확한 주소, 영업시간 등).

### Step 5 — HTML 생성

`rules/html_template.md` 의 골격을 정확히 따라 HTML을 만든다. 핵심:

- **모든 텍스트는 `rules/tone.md` 의 말투로** — 친근체 종결어미, "딱이에요", "다녀왔어요", "진짜", "은근" 등 자연스럽게
- **사진 자리는 `<!-- PHOTO: 설명 -->` 주석으로** — 사용자가 첨부한 사진 개수와 종류에 맞게 배치
- **가운데 정렬 + `<p>&nbsp;</p>` 빈 줄** 활용하여 네이버 특유의 여백감 살리기
- **협찬 마커 처리 (Step 1 결과에 따라 분기)**:
  - `is_sponsored = false` (내돈내산) → 인트로 끝부분에 `#내돈내산` 한 줄 포함
  - `is_sponsored = true` (협찬·체험단) → `#내돈내산` 절대 넣지 말 것. 대신 인트로 시작 직전 또는 가게정보 박스 위에 협찬 명시 문구를 별도 단락으로 넣음. 사용자가 알려준 `sponsor_disclosure` 사용. 사용자가 "기본 문구로"라고 했으면 다음 표준 문구 사용:
    - `소정의 원고료를 받고 작성된 글입니다.` (원고료/광고)
    - `※ 본 포스팅은 ○○으로부터 제품을 제공받아 작성한 후기입니다.` (제품 제공)
- **마무리는 "찾는다면 ~ 방문해보세요" 형식**
- **해시태그 8~15개** — 지역·가게명·지하철역·음식종류·데이트키워드 조합
- 본문 흐름: 표지 → 큰제목 → 인트로 → 가게정보 → 위치 → 주차 → 매장분위기 → 메뉴 → 메뉴별 후기 → 마무리 → 해시태그
- **분량(`min_chars`) 처리**:
  - "분량 제한 없음" → 정보가 있는 만큼 자연스럽게 작성, 억지로 늘리지 않음
  - 숫자 조건이 있으면 → 그 글자수를 **반드시 넘도록** 작성
  - 늘리는 방법: 사실관계를 만들어내지 말고, **각 메뉴에 대한 맛/식감/소스/조합 묘사**, **분위기 디테일**(테이블 배치, 조명, 직원 응대), **방문 팁**(웨이팅·재료소진·예약 방법), **추천 이유 풀어쓰기** 로 자연스럽게 두께를 더한다
  - 같은 표현 반복·미사여구 남발 금지 — `tone.md` 의 친근체 유지하면서 정보 밀도로 채운다

### Step 6 — 파일 저장 + 글자수 측정 + 안내

- 저장 경로: `output/{YYYY-MM-DD}_{slugified_가게명}.html`
  - slugify: 한글은 유지, 공백·특수문자는 `_` 로 변환
- 저장 직후 **본문 글자수를 측정**한다 (Bash로):
  - HTML 태그·HTML 주석·해시태그 라인 제외, 공백 포함
  - 예: `python3 -c "import re,sys; t=open(sys.argv[1]).read(); t=re.sub(r'<!--.*?-->','',t,flags=re.S); t=re.sub(r'<[^>]+>','',t); t=re.sub(r'(?m)^\s*#\S.*$','',t); t=re.sub(r'\s+',' ',t).strip(); print(len(t))" output/{파일명}.html`
- `min_chars` 가 숫자 조건이고 측정 결과가 부족하면 → **사용자에게 보고하지 말고 먼저 본문을 보강해 다시 저장하고 재측정** 한다. 재측정해도 여전히 부족하면 그때 사용자에게 알려서 정보 추가를 요청한다.
- 충족되면 사용자에게 다음 메시지 (글자수 포함):

```
✅ output/{파일명}.html 에 저장했어요.
📏 본문 글자수: {N}자 (최소 {min_chars}자 조건 {충족/제한 없음})

【네이버 블로그에 올리기】
1. 네이버 블로그 글쓰기 → 우측 상단 ⋮ → "HTML 모드" 켜기
   (또는 본문 클릭 후 단축키 사용)
2. 저장된 HTML 파일 내용을 통째로 붙여넣기
3. <!-- PHOTO: ... --> 주석 위치마다 사진 업로드 위젯으로 사진 넣기
4. 미리보기로 확인 후 발행

생성된 HTML을 브라우저에서 미리 열어서 자리 확인하면 편해요:
  open output/{파일명}.html
```

## 사진이 없거나 부족할 때

- 사진을 안 주시면 Step 2 메시지를 다시 한 번 안내
- 5장 미만이어도 "그래도 진행하시겠어요?" 한 번 확인하고 진행 가능
- 사진이 너무 적으면 메뉴별 후기 섹션을 합쳐서 단순화

## 주의

- **rules 파일을 먼저 읽지 않고 바로 작성하지 말 것** — 톤이 어긋남
- **개인 정보 (실제 방문 일시·동행자 이름)는 글에 그대로 안 적기** — 일반화된 표현으로
- **과장된 미사여구·격식체 금지** — `tone.md` 에서 명시한 친근체 유지
- **메뉴판 사진의 가격을 함부로 인용 X** — 가격은 사용자가 알려준 것만 사용 (사진 인식 가격은 참고용)
